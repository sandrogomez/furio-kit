furio-kit — Enterprise Frontend Platform

Author: Sandro A. Gómez Araya  ·  Date: 2026-05-23  ·  Branch: chore/deps-2026-05-13  ·  Status: Production Ready

furio-kit is an enterprise React boilerplate built for FurioLabs-derived frontends. It enforces server-first rendering, explicit layer boundaries, mandatory anti-corruption adapters, and categorical state ownership — all backed by CI and tooling so the architecture is a structural constraint, not a recommendation.

01 Feature-Sliced Design Layer Model

Code is organized into six layers. Imports flow strictly downward. Cross-slice imports at the same layer are forbidden and CI-enforced.

Allowed imports

  • widgets → features, entities, shared
  • features → entities, shared
  • entities → shared
  • views → widgets and below

Forbidden imports

  • Any layer → layer above it
  • entities/user → entities/product
  • features/login → features/cart
  • Anything → @org/ui-kit outside shared/ui

02 App Router Route Groups

Route groups partition the application into authenticated and unauthenticated areas without affecting URL paths. Providers are scoped to authenticated routes only.

03 React Server Components Rendering Model

Every component defaults to a Server Component. "use client" is a boundary directive applied only where hooks, event handlers, or browser APIs are needed. The boundary propagates to all children — push it as deep as possible.

04 Data & State Management

The appropriate tool is determined by the nature of the data, not the feature that uses it. The decision is categorical and enforced by convention.

05 Security Baseline

Security is structural, not advisory. All four pillars are enforced at runtime or build time — not only documented.

06 Auth Architecture & Request Lifecycle

Authentication uses a pluggable adapter pattern. proxy.ts enforces protection on every request via the adapter interface — swapping providers is a one-line change.

Adapter interface

MethodSignatureContract
validateRequest(request: Request) → Promise<AuthUser | null>Read HttpOnly cookie, verify token, return user or null
getLoginUrl(redirectTo?: string) → stringReturn provider login URL with optional post-login redirect

07 Role-Based Access Control

Permissions are defined as a Permission union type. Roles map to permission sets in a static ROLE_PERMISSIONS map. hasPermission() runs server-side; usePermission() runs client-side via Zustand session state.

08 Design System Adapter Pattern

The anti-corruption layer isolates @org/ui-kit behind adapters in src/shared/ui/. The rest of the codebase depends on the adapter's interface, never on the external package's API.

09 Observability

Structured logging and error tracking follow the same adapter pattern as auth. Production-ready implementations are swapped in by replacing the no-op stubs.

ModuleFileDefault behaviorProduction swap
logger shared/observability/logger.ts JSON in production; readable [LEVEL] message in dev Route to Datadog, CloudWatch, etc. via the same interface
errorTracker shared/observability/error-tracker.ts No-op; console.error in dev Replace body with Sentry.captureException / similar
Request ID proxy.ts x-request-id header on every response Forward to log aggregator for request tracing
app/(app)/error.tsx wires the React error boundary to errorTracker.captureException(). All unhandled errors in authenticated routes are captured automatically.

13 Strategic Tradeoff Summary

DecisionPrimary BenefitPrimary Cost
Server-first rendering (RSC default) Smaller JS payload; faster TTFMP; no client data waterfall Steeper mental model; boundary errors are non-obvious
Enforced FSD layer boundaries Predictable dependency graph; safe refactoring; CI-enforced Lateral sharing requires architectural decisions; initial friction
Anti-corruption layer (adapters) Vendor independence; design-system migrations confined to one location Added indirection; ceremony in simple cases
Categorical state ownership Eliminates stale-data bugs; clear tool per concern Occasional state migrations when data nature changes
Security as structural enforcement Cannot ship misconfigured or unauthenticated; fails early Mock adapter blocked in production; env vars required to start
Route group provider scoping Root layout is a pure HTML shell; providers don't leak to public routes Additional layout file per group; mental model of route groups required
Convention enforcement via scaffolding + CI Correct structure every time; onboarding faster Template maintenance; rigid for unusual shapes
